## Задача 1: Исправление web-поиска (КРИТИЧНО)

### Проблема
При включённом web-поиске LLM выдумывает ответ вместо использования результатов поиска. На скриншоте: "Веб-поиск можно включить в браузере, установив Google или Yandex как поисковую систему по умолчанию" — это ответ из головы модели, а не из search-контекста.

### Корневые причины
1. **Системный промпт не упоминает web-поиск** — LLM не знает, что у KnowledgeLab есть встроенный поиск, и интерпретирует вопрос "Web-поиск включен?" как вопрос про настройки браузера
2. **Контекст поиска встраивается в user message** без чёткой инструкции его использовать
3. **Парсер DuckDuckGo хрупкий** — зависит от HTML-структуры

### Решение

**Файл: `scripts/knowledgelab/resources/messages.ru.json`**
- Обновить `local_runtime_system` — добавить:
  - "If the prompt includes a 'Web search context' block, USE those search results to answer the question"
  - "Prioritize web search results over your own knowledge when answering factual questions"
  - "KnowledgeLab has a built-in web search feature that fetches DuckDuckGo results and passes them as context — this is NOT a browser setting"

**Файл: `scripts/knowledgelab/llm/web_search.py`**
- Улучшить `render_web_search_context()` — более чёткая инструкция для LLM:
  ```
  Web search context for the next answer.
  These are REAL search results fetched from the internet for the user's question.
  USE THEM to answer. Do NOT invent answers. If results are relevant, summarize them with URLs.
  If results are irrelevant, say "web-поиск не нашёл релевантных результатов".
  ```
- Добавить JSON API fallback: `api.duckduckgo.com/?q=...&format=json` — стабильнее HTML
- Фильтрация: убрать результаты с `duckduckgo.com` в URL (собственные страницы DDG)
- Приоритет: JSON API → HTML fallback

**Файл: `scripts/knowledgelab/llm/lmstudio.py`**
- В `call_plain_lmstarter()` — убедиться что web search context передаётся в system message (как дополнение), а не только в user message

### Файлы для изменения
1. `scripts/knowledgelab/resources/messages.ru.json` — системный промпт
2. `scripts/knowledgelab/llm/web_search.py` — улучшение контекста + API fallback
3. `scripts/knowledgelab/llm/lmstudio.py` — передача web context в system message

## Задача 2: Адаптация агента под темы пользователя

### Проблема
Системный промпт статичный — LLM не знает, специализируется ли пользователь на React, Unity или чём-то ещё.

### Решение
Добавить `topic_context` в системный промпт на основе текущего маршрута и недавних тем.

**Файл: `scripts/knowledgelab/llm/runtime_context.py`**
- Новая функция `build_topic_context(route, recent_topics) -> str`:
  ```
  Текущая специализация: {scope_description}
  Активные темы: {topic1}, {topic2}, ...
  ```
  - web → "фронтенд/бэкенд разработка, HTML/CSS/JS/React/Vue/Node.js"
  - game → "геймдев, Unity, C#, шейдеры, геймдизайн"
  - general → "общие знания, программирование, референсы"

**Файл: `scripts/knowledgelab/llm/lmstudio.py`**
- Принимать `topic_context` и добавлять в system message

**Файл: `scripts/main.py`**
- В `on_send()` собирать `topic_context` из route и `collect_topic_registry()`
- Передавать в LLM-вызов

**Файл: `scripts/knowledgelab/resources/messages.ru.json`**
- Добавить `topic_descriptions` с описаниями scope

### Файлы для изменения
1. `scripts/knowledgelab/llm/runtime_context.py` — `build_topic_context()`
2. `scripts/knowledgelab/llm/lmstudio.py` — topic_context в system message

## Задача 3: Бесплатная синхронизация vault (Git + Syncthing)

### Проблема
Obsidian Sync стоит денег. Vault — в основном Markdown-текст.

### Решение
Двухуровневая система: Git для бэкапа/истории + Syncthing для быстрого P2P-синка.

**Новый файл: `scripts/knowledgelab/sync/__init__.py`**
- Публичный API

**Новый файл: `scripts/knowledgelab/sync/vault_git.py`**
- `init_vault_git(vault_dir)` — git init + .gitignore
- `vault_git_status(vault_dir) -> dict` — dirty/ahead/behind/untracked
- `vault_git_auto_sync(vault_dir, interval)` — фоновая таска: auto-commit + push/pull

**Новый файл: `scripts/knowledgelab/sync/syncthing.py`**
- `is_syncthing_available() -> bool` — проверка localhost:8384
- `syncthing_status() -> dict` — статус синка

**Файл: `scripts/knowledgelab/config.py`**
- `VAULT_GIT_AUTO_SYNC = False`, `VAULT_GIT_INTERVAL = 300`

**Файл: `scripts/knowledgelab/ui/settings_dialog.py`**
- Секция "Синхронизация vault": git auto-commit, remote URL, статус

**Файл: `scripts/main.py`**
- Запуск auto-sync при старте если включён

### Файлы для создания
1. `scripts/knowledgelab/sync/__init__.py`
2. `scripts/knowledgelab/sync/vault_git.py`
3. `scripts/knowledgelab/sync/syncthing.py`

### Файлы для изменения

## Задача 4: Улучшение транскрипции (regex + LLM)

### Проблема
YouTube-транскрипция без пост-обработки. Нет локальной ASR. Текст содержит повторы, пропущенную пунктуацию, слова-паразиты.

### Решение
Двухуровневая чистка: regex → LLM.

**Новый файл: `scripts/knowledgelab/material/transcript_clean.py`**
- `clean_transcript_regex(text)` — dedup строк, нормализация пунктуации, удаление слов-паразитов, капитализация
- `clean_transcript_llm(text, base_url, model)` — LM Studio постобработка: исправление пунктуации/грамматики, чанкинг по ~2000 токенов

**Файл: `scripts/knowledgelab/material/youtube.py`**
- Вызов `clean_transcript_regex()` после импортаcaption

**Файл: `scripts/knowledgelab/config.py`**
- `TRANSCRIPT_LLM_CLEAN = False` (opt-in)

**Файл: `scripts/knowledgelab/ui/settings_dialog.py`**
- Чекбокс "LLM-постобработка транскриптов"

**Файл: `scripts/sync-youtube-links.py`**
- Вызов clean перед записью в vault

### Файлы для создания
